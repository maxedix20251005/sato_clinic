## 改善アイデア（次フェーズ候補・詳細）
1. News管理の運用性向上
   - 背景: 現状は「保存=反映」に近い運用のため、作業中の内容が意図せず公開状態として扱われるリスクがある。
   - 提案: `news-admin` に `公開 / 下書き` ステータス、`公開開始日`、`公開終了日` を追加し、表示対象を公開中データのみに限定する。
   - 期待効果: 誤公開防止、先行入力（予約投稿）の実現、運用担当者の心理的負担軽減。
   - 実装観点: `scripts/news-admin.js` のデータ構造拡張、`scripts/news-live.js` の表示フィルタ追加、管理UIの入力欄追加。

2. 予約フォームの実運用強化（空き枠の実在庫化）
   - 背景: 現状の ○△× はフロント内ロジック中心で、実際の予約枠在庫とは連動していない。
   - 提案: 予約枠マスタを外部データ（Google Spreadsheet/Calendar など）で管理し、画面上の空き表示と同期待ち件数を同期する。
   - 期待効果: ダブルブッキング抑止、受付オペレーションとの整合、手動調整コスト削減。
   - 実装観点: GitHub Pages前提ではサーバレス連携（Apps Script API等）を利用。通信失敗時のフォールバック表示も定義する。

3. 多言語品質の底上げ（医療用語対訳の標準化）
   - 背景: ページ追加・文言更新時に都度翻訳すると、語彙やトーンがページ間で揺れやすい。
   - 提案: `docs/` 配下に「医療用語対訳集（JA/EN）」と「表記ルール（敬体・単数複数・用語優先度）」を作成し、更新時の参照元を一本化する。
   - 期待効果: 翻訳品質の均一化、レビュー時間短縮、医療文脈での誤訳抑止。
   - 実装観点: `AI_BUILD_PROMPT.md` と `TECH_SPEC_NEW.md` から参照する運用ルールとして明文化する。

4. 監視と障害検知（フォーム送信・主要導線）
   - 背景: 現状は不具合発生時に利用者報告で気づく受け身運用になりやすい。
   - 提案: Formspree送信結果（失敗率/急増）と主要導線（予約ページ到達、送信完了到達）を定期確認できる簡易監視手順を整備する。
   - 期待効果: 障害の早期発見、影響範囲の可視化、運用判断の迅速化。
   - 実装観点: 外部分析ツール導入前でも、まずは週次点検チェックリスト + 手動ログ確認ルールを `docs/` に定義する。

5. 品質保証フローの自動化（回帰チェック最小セット）
   - 背景: ページ・文言改修が増えると、リンク切れや言語切替漏れ等の回帰が再発しやすい。
   - 提案: 最低限の自動チェックを導入（主要リンク生存、予約/問診の必須入力、言語切替、news-admin保存→反映）。
   - 期待効果: リリース前検証の抜け漏れ削減、確認工数の平準化、品質の再現性向上。
   - 実装観点: 初期は手軽なスクリプト（例: Playwrightのスモークテスト）から開始し、失敗時の再現手順を `WIP` と `NEWS_UPDATE_WORKFLOW` に連携する。
